# Agentic RAG System

> Intelligent document Q&A with agentic reasoning — decides how to retrieve, checks quality, falls back to web search, and reflects on answers.

Built with Node.js, LangGraph.js, Qdrant, and Groq LLaMA 3.3 70B. No OpenAI dependency — fully local embeddings, free inference.

---

## Architecture

### Phase 2 — Advanced Retrieval Pipeline
```
Query
  → Multi-Query Generation (3 variants via Groq LLaMA 3.3 70B)
  → HyDE Embedding (hypothetical answer → embed instead of raw question)
  → Dense Search (Qdrant cosine similarity, top-10, 384-dim)
  → BM25 Sparse Search (Qdrant sparse vectors, keyword matching, top-10)
  → RRF Fusion (Reciprocal Rank Fusion, k=60, rank-based merge)
  → Cohere Rerank (cross-encoder rerank-english-v3.0, top-4)
  → Parent Deduplication (unique context per parent chunk only)
  → Groq LLaMA 3.3 70B → Grounded Answer
```

### Phase 3 — Agentic Layer (LangGraph.js)
```
Query
  → Query Router (classify: factual / analytical / out-of-scope)
  → Retrieval Node (full Phase 2 pipeline)
  → Grading Node (Groq evaluates: are retrieved docs relevant? yes/no)
      → if NO → CRAG loop → Tavily web search fallback
      → if YES → Generation Node
  → Self-RAG Reflection Node (Groq scores: is answer faithful? complete?)
      → if NO → regenerate with refined context
      → if YES → return final answer
  → LangSmith Tracing (every node logged, full observability)
```

## Key Techniques Implemented

### Retrieval
- **Parent-child chunking** — child chunks (150 chars) used for retrieval precision, parent chunks (500 chars) sent to LLM for full context. Search small, read big.
- **Hybrid search** — BM25 sparse vectors + dense semantic embeddings running simultaneously in Qdrant. Captures both exact keyword matches and semantic meaning.
- **Reciprocal Rank Fusion (RRF)** — rank-based fusion of dense and sparse results (k=60). Eliminates score scale mismatch between cosine similarity and TF-IDF distributions.
- **HyDE (Hypothetical Document Embeddings)** — Groq generates a hypothetical answer to the question, that answer is embedded instead of the raw question. Hypothesis lives closer to real chunks in embedding space → higher retrieval scores.
- **Multi-query retrieval** — 3 query variants generated by Groq, each embedded with HyDE, results merged. Different phrasings surface chunks that a single query misses.
- **Cohere cross-encoder reranking** — takes query + each candidate chunk as joint input, full transformer attention over both. Scores true relevance, not vector geometry approximation.

### Agentic
- **CRAG (Corrective RAG)** — retrieval evaluator grades doc relevance. Low score → Tavily web search fallback. High score → direct generation. Based on arxiv 2401.15884.
- **Self-RAG reflection** — after generation, reflection node scores faithfulness and completeness. Low score → regenerate. Based on arxiv 2310.11511.
- **Query router** — classifies query type (factual / analytical / out-of-scope) at runtime and routes to appropriate retrieval strategy. Simple queries skip multi-query + HyDE to save latency.
- **Conditional edges** — LangGraph graph edges evaluate state at runtime and branch accordingly. Not a fixed pipeline — decisions made per query.

## API Reference

### POST /ingest
Ingest a PDF into the vector store with parent-child chunking.

**Request:**
```json
{ "filePath": "C:/path/to/document.pdf" }
```

**Response:**
```json
{
  "status": "ok",
  "pages": 1,
  "chunks": 19,
  "pointsStored": 19
}
```

### POST /query
Ask a question. Runs full retrieval pipeline → agentic loop → grounded answer.

**Request:**
```json
{ "question": "How does the reversal algorithm work?" }
```

**Response:**
```json
{
  "answer": "The reversal algorithm works by...",
  "retrieved": 3,
  "sources": [
    {
      "childText": "void reverseArray(int arr[], int start, int end){...",
      "parentText": "4. REVERSAL ALGORITHM (OPTIMAL)...",
      "rerankScore": 0.9241,
      "parentIndex": 3,
      "source": "notes.pdf"
    }
  ]
}
```

### GET /debug
Inspect current vector store state.

```json
{
  "status": "ok",
  "totalPoints": 19,
  "vectorSize": 384,
  "sampleChunks": [{ "id": 755338925, "preview": "ARRAY ROTATION — COMPLETE NOTES..." }]
}
```

### GET /benchmark
Compare naive vs hybrid vs full pipeline across test questions.

```json
{
  "status": "ok",
  "benchmark": [
    {
      "question": "How does the reversal algorithm work?",
      "strategies": {
        "naive":  { "top1Score": 0.6594, "top1Preview": "..." },
        "hybrid": { "top1RRFScore": 0.0301, "top1Preview": "..." },
        "full":   { "top1RerankScore": 0.9241, "top1Preview": "..." }
      }
    }
  ]
}
```

### DELETE /reset
Wipe and recreate Qdrant collection.

### Prerequisites
- Node.js v18+
- Docker Desktop

### Install
```bash
git clone https://github.com/SHUBHANSHU602/agetic-rag
cd agetic_rag
npm install
cp .env.example .env
# Fill in your keys
```

```
PORT=5000
GROQ_API_KEY=your_groq_key
COHERE_API_KEY=your_cohere_key
TAVILY_API_KEY=your_tavily_key
LANGSMITH_API_KEY=your_langsmith_key
LANGSMITH_PROJECT=agentic-rag
```

### Run
```bash
# Terminal 1 — Qdrant
docker run -p 6333:6333 qdrant/qdrant

# Terminal 2 — Server
nodemon index.js
```

### Docker Compose (Production)
```bash
docker-compose up
```

## Design Decisions

**Why @xenova/transformers instead of OpenAI embeddings?**
No API key, no cost, runs fully local. Tradeoff is lower representational power — MiniLM is distilled from a larger teacher model so embedding geometry is flatter. Score threshold set to 0.05 accordingly (vs typical 0.3 for cloud embeddings).

**Why pdf2json instead of pdf-parse or pdfjs-dist?**
Both pdf-parse and pdfjs-dist failed on Node.js v24 due to native dependency issues. pdf2json works reliably. Handles encoding errors with try/catch fallback on decodeURIComponent.

**Why RRF instead of weighted score combination?**
Dense scores (cosine similarity) and sparse scores (TF-IDF normalized) have incompatible distributions. Directly comparing them is meaningless. RRF uses only rank position, eliminating the scale mismatch entirely.

**Why parent-child chunking?**
Small child chunks (150 chars) give precise retrieval — the right fragment gets matched. Large parent chunks (500 chars) give full context to the LLM. Single-size chunking forces a tradeoff between precision and context; parent-child eliminates it.

**Why a grading node before generation?**
Without grading, irrelevant chunks pass to the LLM which then hallucinates a coherent-sounding but wrong answer. The grading node is a quality gate — if retrieved docs score low, the system falls back to web search (CRAG) instead of generating from bad context.
